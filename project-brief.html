Korean Conversation Practice

Project brief & wireframes — personal learning tool built around Active Korean workbooks

1. Project Brief

What it does

A local web app that lets me practice Korean conversational patterns from Seoul National University's Active Korean series. I pick a scenario (e.g. "introduce yourself", "order at a restaurant"), get assigned a randomized role, and have a spoken conversation with an AI partner.

Architecture

A single Python process serves both the frontend and API. Launch with one command (uv run start), opens a browser tab to localhost.

Browser (Frontend)
HTML / CSS / vanilla JS MediaRecorder API — audio capture SpeechSynthesis API — TTS via Yuna (Premium) Hold-to-talk keyboard events
HTTP & SSE ↓↑
Python Backend (FastAPI)
Serves static HTML/JS POST /api/transcribe — receives audio, runs whisper-cli POST /api/chat — sends learner text to agent, streams response GET /api/scenarios — lists available scenarios
↓↑
Claude Agent SDK
Multi-turn conversation via ClaudeSDKClient Custom tools: speak(text), correct(hint) Routed through Claude Max subscription

Key components

  • Frontend — plain HTML/CSS/JS served by the backend. No build step, no framework.
  • FastAPI backend — async Python web framework. Serves the page and handles API endpoints. Async-native, which matches the Claude Agent SDK's async API.
  • Claude Agent SDK (claude-agent-sdk) — drives the conversation via multi-turn sessions, routed through Claude Max subscription (no metered API costs)
  • TTS — browser's SpeechSynthesis API using the Yuna (Premium) Korean voice. Runs entirely client-side, no backend call needed.
  • STT — browser records audio via MediaRecorder, sends the blob to the backend, which runs whisper-cli (whisper.cpp) and returns Hangul text
  • uv for virtualenv/dependency management, pyproject.toml for portability

Why a web app instead of pygame?

  • Self-verification — Chrome DevTools MCP lets the development agent (Claude Code) take snapshots, click elements, check console errors, and verify the app end-to-end without manual testing
  • TTS is free — browser's SpeechSynthesis API with Yuna Premium, no subprocess call needed
  • Audio recording — MediaRecorder API handles it natively
  • Still feels like one app — single uv run start command, localhost only, no deployment concerns

Voice input: quasimodal hold-to-talk

  • Hold a key (e.g. Space) to record — browser captures audio via MediaRecorder
  • Press ESC while holding to cancel and start over
  • Release the key — audio blob is sent to POST /api/transcribe
  • Backend runs whisper-cli, returns Hangul text, which is sent to the conversation agent

Teaching approach

The agent uses Socratic correction: if I make a mistake, it never gives me the answer directly. Instead it gives hints — pointing out which particle is wrong, reminding me of a grammar rule, or asking me to try again. The agent has two output channels (implemented as custom tools):

  • speak(text) — text to say aloud as the conversation partner. Frontend receives this via SSE and passes it to SpeechSynthesis.
  • correct(hint) — written feedback/hints shown on screen when I make a mistake

4. Template Format: YAML vs Python

The templates above need to be defined somewhere in code. Two natural choices: YAML data files with a custom resolver, or Python classes. Here's the same template (Unit 2 self-introduction) in both formats, followed by a tradeoff analysis.

YAML

scenario: self-introduction
unit: 2
title: Self-Introduction at a Party
grammar:
  - topic particles (은/는)
  - 이다 copula

context:
  name:
    pool: korean_names
  country:
    pool: unit2.countries
  occupation:
    pool: unit2.occupations

system_prompt: |
  You are a friendly Korean speaker at a
  party. The learner's name is {name}.
  They are from {country} and work as
  a {occupation}. Ask about their name,
  nationality, and occupation in order.

  Only use grammar and vocabulary from
  Active Korean Units 1-2. If they make
  a mistake, use the correct tool to
  give a Socratic hint.

flow:
  - role: partner
    intent: greet_and_ask_name
  - role: learner
    intent: introduce_name
    grammar: [topic particle, 이다]
  - role: partner
    intent: ask_nationality
  - role: learner
    intent: state_nationality
  - role: partner
    intent: ask_occupation
  - role: learner
    intent: state_occupation
  - role: partner
    intent: closing

Python

from scenarios import Scenario
import random

COUNTRIES = [
    "미국", "중국", "일본", "인도",
    "호주", "영국", "독일", "프랑스",
    "캐나다", "한국", "러시아",
]

OCCUPATIONS = [
    "선생님", "학생", "의사", "요리사",
    "은행원", "기자", "회사원", "연구원",
]

class SelfIntroduction(Scenario):
    unit = 2
    title = "Self-Introduction at a Party"
    grammar = [
        "topic particles (은/는)",
        "이다 copula",
    ]

    def setup(self):
        self.name = random_korean_name()
        self.country = random.choice(
            COUNTRIES
        )
        self.occupation = random.choice(
            OCCUPATIONS
        )

    def system_prompt(self):
        return f"""You are a friendly
Korean speaker at a party. The learner's
name is {self.name}. They are from
{self.country} and work as a
{self.occupation}. Ask about their name,
nationality, and occupation in order.

Only use grammar and vocabulary from
Active Korean Units 1-2. If they make
a mistake, use the correct tool to give
a Socratic hint."""

Tradeoff Comparison

DimensionYAMLPython
Readability Very readable, declarative, scannable Readable but more verbose
Randomization Needs a custom resolver engine to handle {pool: ...} references Native — just call random.choice()
Validation Runtime only; typos in pool names or field names fail silently or late IDE autocomplete, type checking, errors caught immediately
Flexibility Limited to what the YAML engine supports; conditional flows (e.g. "friend available" branching in phone call) need custom syntax Arbitrary logic — if/else, loops, conditional prompts all natural
System prompt String templates with {var} — needs a template engine to resolve f-strings with full Python expressions
Dependencies Requires PyYAML No extra dependency
Separation of concerns Data is separate from code Data and logic co-located in one file
Adding a new scenario Copy a YAML file, edit fields Write a new class (a few more lines, but with guidance from base class)
Who edits them Anyone — no Python knowledge needed Requires Python familiarity (but it's just you)
Flow definition Clean list of steps — but intent strings are just labels with no enforcement Flow can be implicit in the system prompt, or explicit as a list — either way, logic is enforceable
Recommendation: Python classes.

For a personal project where you're the only author:

  1. Randomization is first-class — no custom YAML resolver needed, just random.choice()
  2. System prompts need nuance — they'll reference grammar rules, vocabulary constraints, correction styles. f-strings and methods handle this naturally; YAML string templates get awkward fast
  3. Conditional flows — the phone call template has branching (friend available vs. busy). In YAML you'd need to invent control-flow syntax; in Python it's just if/else
  4. No extra dependency — no PyYAML needed
  5. IDE support — typos in vocabulary lists, missing fields on a scenario class, wrong method signatures — all caught before runtime

The one advantage of YAML — separation of data from code — matters when non-programmers edit templates. Since this is a personal project, that advantage doesn't apply.
